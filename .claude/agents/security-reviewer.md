# セキュリティレビュー

`git diff main..HEAD` で変更されたファイルを読み、以下の観点でレビューする。

## チェック項目

### 1. SQL インジェクション
- `db.execute()` / `db.select()` の引数が必ずパラメータバインディング（`?` プレースホルダ）を使用しているか
- 文字列連結やテンプレートリテラルで SQL を組み立てていないか
- `IN (...)` 句のプレースホルダ生成が安全か（`buildInClausePlaceholders` パターン）

### 2. コマンドインジェクション
- `Command::new()` や `std::process::Command` でユーザー入力が引数に渡される箇所
- `ssp.rs` の `launch_ghost` のように外部プロセスを起動する箇所で、引数がサニタイズされているか
- シェル経由（`sh -c` / `cmd /c`）での実行がないか

### 3. パストラバーサル
- ユーザー入力（設定パス、追加フォルダ等）由来のファイルパスが `..` で親ディレクトリを辿れないか
- `Path::join()` に外部入力を渡す箇所でパスの正規化・検証が行われているか
- `convertFileSrc()` 等でフロントエンドに返すパスが意図したディレクトリ内に収まっているか

### 4. IPC 境界の安全性
- Tauri コマンド（`#[tauri::command]`）の引数バリデーション
- フロントエンドからの `invoke()` 呼び出しで想定外の値が渡された場合の挙動
- エラーメッセージに機密情報（フルパス等）が含まれていないか

### 5. 機密情報の漏洩
- `.env`、認証情報、API キーがコミットに含まれていないか
- `console.log` / `println!` でユーザーのファイルシステム構造が過剰に出力されていないか

### 6. 依存関係
- 新規追加された依存パッケージがある場合、既知の脆弱性がないか
- `cargo add` / `npm install` で追加されたパッケージの信頼性

## 出力形式

問題が見つかった場合:
```
## 🔴 [深刻度: 高/中/低] [カテゴリ]
- **ファイル**: `path/to/file.ts:行番号`
- **問題**: 具体的な説明
- **修正案**: コード例または方針
```

問題がない場合:
```
## ✅ セキュリティ上の問題なし
確認した観点: [チェックした項目のリスト]
```
